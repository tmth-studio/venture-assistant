# Customer Transformation Model (CTM)

Source: IVE Systems / reflect.site CTM guide

---

## What It Is

The CTM is a customer-centric view of how the venture works — akin to a customer journey map. It shows how the 4-D Product (Working, Communication, Payment, Partner) creates the necessary shifts in customers' cognitive, emotional, and behavioural states as they advance through their purchasing experience.

Where the Ops Map shows how the venture produces and delivers its products, the CTM shows what those products must achieve in the customer — the state changes required for the venture to function.

---

## Four Journey Phases

Every customer must move through four phases for the venture to work at scale:

1. **Becoming a customer** — moving from unaware (experiencing a Key Cost) to convinced and signed up
2. **Consuming / using the product** — using the product effectively enough to realise maximum value
3. **Making payment** — completing the payment transaction at the right moment
4. **Repeating** — returning for the next purchase cycle

---

## State Change Framework

At each phase transition, specify what the customer must:

| Dimension | Question |
|-----------|----------|
| **Cognitive** | What must they *think*? |
| **Emotive** | What must they *feel*? |
| **Behavioural** | What must they *do*? |

Then assign which 4-D Product drives that transformation:
- **Working Product** — core functionality
- **Communication Product** — awareness, conversion, activation
- **Payment Product** — getting paid, collections, defaults
- **Partner Product** — gateway partner onboarding and integration

---

## 4-Step Construction Process

### Step 1 — Define Acquisition State Changes
Map the journey from initial state (unaware, experiencing Key Cost) to becoming a customer. This phase is almost always driven by a Communication Product.

*Example (Calmly — dispute resolution):* Claimants start unaware of the service, experiencing the key cost of an unresolved dispute. Required state change: "Calmly can help me + I become a customer."

### Step 2 — Map Consumption State Changes
For each stage of product use, define the state changes needed for maximum value realisation and assign the Working Product components that drive them.

### Step 3 — Insert Payment
Place the Payment Product at the appropriate point in the journey. Start with standard industry payment practice, then optimise for strategic placement (informed by R6 work).

### Step 4 — Review
Check the full CTM for:
- **Logical viability** — do the state transitions follow naturally from each other?
- **Realistic progressions** — is each state change achievable given where the customer starts?
- **Elegance** — can any steps be collapsed or driven by a single product component?
- **Strategic operationalisation** — does the CTM reflect the BALM solutions, not just conventional practice?

---

## Design Principle: Avoid Overdesign

> "One of the biggest mistakes you can make at this stage is to overdesign."

Start with the minimum viable CTM. Specify the minimum level possible — adopt standard industry practices until a requirement demands otherwise. The CTM evolves as BALM requirements are productized. A CTM designed before the requirements are solved will bake in the conventional CBA.

---

## Relationship to Ops Map and BALM

| Tool | What it shows |
|------|--------------|
| **Ops Map** | How the venture produces and delivers the four products |
| **CTM** | What those products must achieve in the customer |
| **BALM** | What constraints the products must satisfy to be viable at scale |

The CTM and Ops Map are complementary views of the same system. The Ops Map is production-oriented (inside-out); the CTM is customer-oriented (outside-in). Both must be consistent — if the CTM requires a state change that the Ops Map cannot produce affordably, there is an unresolved BALM requirement.

---

## Using the CTM in Modelling

When building the operational and financial model:

1. Build the CTM for each customer type (claimant, law firm as gateway, etc.)
2. Each state change in the CTM implies a product component in the Ops Map
3. Each product component implies a resource (personnel, durable, consumable)
4. Each resource has a cost
5. Summed across all state changes → full cost structure

The CTM makes visible the full set of product components the venture must produce — including the ones that are easy to miss because they serve indirect customers (gateway partners, secondary users).
